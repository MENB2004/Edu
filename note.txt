// Introduction to Git
Git is a distributed version control system that tracks changes in any set of computer files, usually used for coordinating work among programmers collaboratively developing source code during software development. Its goals include speed, data integrity, and support for distributed, non-linear workflows.

**Key Concepts:**

*   **Repository (Repo):** A directory where Git stores all the project files and the complete history of changes.
*   **Commit:** A snapshot of your repository at a specific point in time. Each commit has a unique ID (SHA-1 hash), a message, an author, and a timestamp.
*   **Branch:** A lightweight movable pointer to a commit. It allows you to diverge from the main line of development and work on new features or bug fixes independently.
*   **Master/Main:** The default branch name in Git, representing the main line of development.
*   **Head:** A pointer to the current branch you are working on.
*   **Working Directory:** The files you see and edit on your local machine.
*   **Staging Area (Index):** An intermediate area where you prepare changes before committing them.
*   **Remote Repository:** A version of your repository hosted on a server (e.g., GitHub, GitLab, Bitbucket) that allows for collaboration and backup.

**Basic Git Workflow:**

1.  **Initialize a repository:** `git init`
2.  **Add changes to the staging area:** `git add <file>` or `git add .` (for all changes)
3.  **Commit changes:** `git commit -m "Your commit message"`
4.  **View history:** `git log`
5.  **Create a new branch:** `git branch <branch-name>`
6.  **Switch to a branch:** `git checkout <branch-name>` or `git switch <branch-name>` (newer)
7.  **Merge branches:** `git merge <branch-to-merge>`
8.  **Push to a remote repository:** `git push <remote-name> <branch-name>`
9.  **Pull from a remote repository:** `git pull <remote-name> <branch-name>`

**Common Git Commands:**

*   `git status`: Shows the status of your working directory and staging area.
*   `git diff`: Shows changes between the working directory and the staging area, or between commits.

*   `git rebase <base-branch>`: Rewrites commit history by moving or combining commits.

**How to use Github for version & collaboration**

GitHub is a web-based platform that uses Git for version control. It provides a centralized place for developers to host and manage their Git repositories, facilitating collaboration, code sharing, and project management.

**Key Features of GitHub:**

*   **Repository Hosting:** Stores Git repositories online, making them accessible from anywhere.
*   **Collaboration Tools:**
    *   **Pull Requests (PRs):** A mechanism for proposing changes to a repository. Developers can discuss, review, and approve changes before merging them into the main codebase.
    *   **Code Review:** Allows team members to review each other's code, provide feedback, and ensure code quality.
    *   **Issues:** A system for tracking bugs, feature requests, and other tasks related to a project.
    *   **Wikis:** For project documentation.
    *   **Projects (Kanban boards):** For task management and workflow visualization.
*   **Social Coding:** Features like starring repositories, following users, and contributing to open-source projects.
*   **Integrations:** Connects with various third-party tools for continuous integration/continuous deployment (CI/CD), project management, and more.

**Basic GitHub Workflow for Collaboration:**

1.  **Fork a Repository (for open-source contributions) or Clone a Repository (for team projects):**
    *   **Fork:** Creates a personal copy of a repository on your GitHub account. You make changes in your fork and then propose them back to the original repository via a Pull Request.
    *   **Clone:** Downloads a copy of a repository to your local machine. You typically clone repositories you have direct write access to (e.g., team projects).
        `git clone <repository-url>`

2.  **Create a New Branch:**
    *   Always work on a new branch for new features or bug fixes to keep the `main` branch clean.
        `git checkout -b <feature-branch-name>`

3.  **Make Changes and Commit:**
    *   Edit files, add new code, etc.
    *   Stage your changes: `git add .`
    *   Commit your changes: `git commit -m "Descriptive commit message"`

4.  **Push Changes to Your Remote Branch:**
    *   Upload your local branch to your GitHub repository.
        `git push origin <feature-branch-name>`

5.  **Create a Pull Request (PR):
    *   Go to your repository on GitHub.
    *   GitHub will usually detect your newly pushed branch and offer a "Compare & pull request" button.
    *   Fill in the PR title and description, explaining what changes you made and why.
    *   Assign reviewers (if applicable) and link to any relevant issues.

6.  **Code Review and Discussion:**
    *   Team members or maintainers review your code, suggest changes, and discuss the implementation.
    *   You might need to make further commits to your branch and push them to update the PR.

7.  **Merge Pull Request:**
    *   Once the code is approved, a maintainer merges your branch into the target branch (e.g., `main`).
    *   After merging, you can delete your feature branch on GitHub and locally.

8.  **Sync Your Local Repository:**
    *   After a PR is merged, pull the latest changes from the remote `main` branch to keep your local repository up-to-date.
        `git checkout main`
        `git pull origin main`
